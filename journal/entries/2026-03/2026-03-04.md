### Summary
PRD #3 was closed as infeasible after discovering that Datadog's managed settings override all user-level OpenTelemetry configuration. The developer investigated Claude Code's native OTEL support, found it emits 8 metrics and 5 event types (via the logs protocol), and attempted to route that telemetry to a local Datadog Agent running on the same machine as cluster-whisperer. However, managed settings (MDM-distributed by Datadog as the developer's employer) forcibly redirect all telemetry to an internal endpoint (`ai-devx-api.us1.prod.dog`) with higher precedence than any user settings, making it impossible to send data to the local agent or a personal Datadog org.

The developer installed the official Datadog MCP server (a remote HTTP endpoint with OAuth authentication, not an npm package) to verify telemetry flow, which revealed the managed settings override. They also fixed a `/make-autonomous` issue where the `prds-get` skill lacked a YOLO variant, causing symlink confusion in other projects. Both the MCP server addition and the YOLO variant fix were merged to main via PR #27.

Before closing the PRD, the developer reverted the OTEL environment variables from `~/.claude/settings.json` (since routing to the local agent was now impossible), updated the research documentation in the advocacy-skills repo with findings about managed settings, and created 5 Anki cards capturing the key learnings: the two-signal-type structure of Claude Code's telemetry, the gotcha that events use the logs protocol rather than metrics, the managed settings override that killed the PRD, the nature of the official Datadog MCP server, and the hardcoded service name that separates Claude Code data from other services.

The PRD was then archived to `prds/done/` with status "Closed — No Longer Feasible" and the closure reason documented. The GitHub issue was updated and closed with a detailed explanation of the managed settings blocker.
