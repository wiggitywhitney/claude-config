### Summary
The developer created a product requirements document for a global cluster lifecycle reminder hook that replaces mandatory teardown rules with awareness-based notifications.

The work started from a real problem: the developer had cluster teardown requirements baked into multiple project-level rules, but those clusters were often legitimately needed across multiple sessions. Forcing teardown at project boundaries created friction, and relying on manual reminders meant forgotten clusters could rack up costs (GKE clusters run about $0.19-$0.57 per hour).

The developer's initial idea was to add a SessionEnd hook that would detect running clusters and remind the user whether to tear them down or keep them running. After exploring the codebases in cluster-whisperer and kubecon-2026-gitops, the approach shifted: a SessionStart hook made more sense because it catches forgotten clusters right when the developer starts a new session and can actually act on them, whereas SessionEnd hooks are cleanup-only and can't show messages to the user.

The PRD lays out the full strategy. The solution has three main parts: a new script that detects both Kind and GKE clusters, wiring that script as a global hook in the Claude settings, and softening the infrastructure safety rules across the affected projects to remove mandatory teardown language. The document includes six milestones covering script creation, hook setup, rule updates, testing, and end-to-end verification.

The design handles both cluster types with appropriate urgency levels—Kind clusters are free but consume local resources, while GKE clusters cost money, so the reminder differentiates between them. The script gracefully skips checks if the tools aren't installed, and includes specific teardown command hints for each cluster type. The hook runs globally rather than per-project because forgotten clusters cost money regardless of which repository the developer is working in.

The tests use stub executables to simulate `kind` and `gcloud` without requiring actual installations, and verify that output is valid JSON with the expected `additionalContext` field.

The developer also updated the PRD status from Draft to In Progress, marked milestones M1 and M5 complete, and documented the work in the project's progress log and daily summary. The remaining work includes wiring the script as a global SessionStart hook, softening infrastructure safety rules, removing duplicate teardown milestones from other PRDs, and end-to-end verification with a running cluster.
