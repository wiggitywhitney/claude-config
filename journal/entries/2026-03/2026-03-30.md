### Summary
The developer created a PRD for migrating six git-related hooks from Claude Code's tool-call interception layer to native git hooks, providing deterministic process-level enforcement that's harder for an AI agent to circumvent.

The work started with a conversation about Claude plugins—specifically the Skill Creator, Superpowers, and Code Review plugins. The developer wanted to understand how those plugins were constructed, with an eye toward upgrading their own frequently-used skills (PRD skills, advocacy checks) and creating Anki cards to learn the concepts deeply. That exploration led to a question from a friend: can Claude Code jailbreak its own hooks? The developer asked for research on this, which became the foundation for the entire PRD.

The research showed that while Claude Code hooks are more reliable than CLAUDE.md instructions, they can still be behaviorally circumvented through creative command routing and regex gaps. Native git hooks operate at a fundamentally different layer—inside the git process itself—making them much harder to bypass. The developer saved this research to a separate file and referenced it in the PRD.

The core decision was to do a clean migration rather than dual enforcement. The developer initially questioned why to keep both systems, but after discussing the tradeoffs, settled on a clean approach: native git hooks handle all blocking and messaging, Claude Code hooks for migrated checks get archived, and two hooks that intercept non-git commands (`gh pr create` and `gh pr merge`) stay as Claude Code hooks since there's no git hook equivalent.

A critical requirement emerged during the conversation: every hook must print smart, actionable error messages that both humans and Claude can understand and act on. The developer emphasized this should be in the success criteria for all milestones, not just mentioned in passing.

The PRD was structured around five milestones: infrastructure setup with a bootstrap installer, migration of four "clean" hooks (commit message checks, branch protection, progress tracking, test tiers), migration of two "complex" hooks (build/typecheck/lint verification and security checks), rollout across active repos, and future exploration of plugin architecture patterns.

During planning, the developer discovered their corporate laptop has a globally-set `core.hooksPath` pointing to Datadog's enterprise hook system. Rather than treat this as a blocker, they incorporated it into the architecture—Datadog's hooks dispatch to local `.git/hooks/` via a `run-local-hooks` mechanism, which means the new native hooks will work within that existing system without conflicts. The developer was careful to investigate what was already there and designed the migration to preserve the existing commit-story journaling hooks that depend on this system.
