## 8:57:38 AM PDT - Commit: 15fab6b

### Summary
The developer fixed three issues in a Git hook script that manages progress file updates for pull requests, addressing findings from a code review.

The first fix corrected how the script counts commits between branches. It switched from a three-dot range (`BASE_REF...HEAD`) to a two-dot range (`BASE_REF..HEAD`). The three-dot syntax counts commits reachable from either side, which can produce a non-zero count even when a branch has no new commits—causing false positives. The two-dot range correctly counts only commits unique to the branch being pushed.

The second fix scoped where new entries get inserted in the progress file. Previously, the script searched the entire file for an `### Added` section and inserted there, which could accidentally add entries to historical release sections. Now it first locates the `## [Unreleased]` section, then searches for `### Added` only within that section. If the section exists but has no `### Added` subsection, the script creates one. If there's no `## [Unreleased]` section at all, it falls back to appending at the end of the file.

The third fix tightened validation on the auto-draft content. The script now blocks the accept path if the auto-draft is empty or contains only whitespace, preventing empty entries from being committed. The edit path was also updated to use the same whitespace-trimmed check, ensuring consistent behavior between the two code paths.
