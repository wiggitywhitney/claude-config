## Key Decisions

- **Propagate decisions to downstream milestones after recording in decision log.** Decisions were becoming "dead" context—future agents read milestone descriptions but didn't necessarily check the decision log. Milestone updates reference the decision to keep the log as the single source of truth while making sure work instructions stay current.

- **Consolidate prd-update-decisions process overview from 7 to 5 steps.** Process overview listed 7 steps but the document contained only 5 Step sections. Merged related steps (e.g., "Identify Decision Points" and "Map to PRD Sections" became "Assess Decision Impact") to eliminate redundancy and match actual structure.

- **Place milestone update notes inline, not appended separately.** Agents were adding notes inconsistently. Specified that notes should sit next to the affected content (next to changed acceptance criteria or at top of milestone description for broad changes).

- **Clarify distinction between new milestone items and stale content updates in propagation rules.** New items added during decision logging are additions; propagation specifically addresses updating existing milestone content that's become outdated due to decisions.

- **Specify concrete format for propagation report output.** Replaced vague language ("briefly tell the user") with a specific example format showing list structure with decision references.

- **Add decision awareness nudges to prd-update-progress and research skills.** Don't make it a blocking requirement—keep it as a nudge to the agent so it preserves autonomy. The agent assesses whether decisions emerged and runs prd-update-decisions itself if needed.

- **Target global CLAUDE.md at under 150 lines, ideally ~120.** Every line gets loaded into every conversation, creating direct token cost. 150 is the hard constraint; 120 provides headroom for future additions.

- **Keep domain-specific short sections inline rather than extracting.** OpenTelemetry Packaging (6 lines), Vals Secrets Management (4 lines), and ABOUTME File Headers (3 lines) are already concise. Extracting them would add file overhead without meaningful savings.

- **Extract verbose sections into @-referenced rule files.** Testing, Git Workflow, Rules Enforced by Hooks, Issue Juggling, Adopting New Technologies, Datadog Enterprise Environment, and PRD Workflow sections get moved to separate files with projected savings of ~60 lines.

- **Preserve existing wording verbatim when extracting content to rule files.** Existing sections have been carefully worded through iterations. Extract as copy-paste rather than paraphrasing to maintain intentional phrasing.

- **Keep CLAUDE.md as a concise index with 1-2 line summaries plus @path/to/file references.** CLAUDE.md must remain self-contained enough that a fresh agent understands rules without reading every referenced file. Index lines must be actionable, not just pointers.

- **Standardize reference paths to symlink-based pattern (@~/.claude/rules/).** Replaces verbose full paths for cleaner, more maintainable documentation.
