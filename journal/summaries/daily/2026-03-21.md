# Daily Summary — 2026-03-21

## Narrative

The developer spent the day fixing a series of interconnected issues in the git hook system that were falsely blocking commits and pushes, then wrapped up with configuration and documentation cleanup.

The morning started with a critical bug: repositories with large test suites (1700+ tests generating 20KB+ of output) were being blocked by git hooks even though all tests actually passed. The root cause was that shell command substitution using `$()` was producing incorrect exit codes when capturing large amounts of output. The developer fixed this by replacing the pipe capture pattern with temporary file redirection across three hook scripts, decoupling output capture from exit code capture. They also removed the `pipefail` flag from verify-phase.sh since it was interfering with eval'd commands that contain implicit pipelines. A detailed test suite with 20 tests validated the fix across small, medium, and very large output scenarios.

But that wasn't the end of it. Even after the temp file fix, hooks were still blocking valid commits. The developer discovered that external processes—specifically a GitLab credential helper prompt—were interfering with exit code capture during hook execution. Rather than trying to prevent the corruption, they added a defensive override: when the process exit code is non-zero but the output contains a special marker indicating the tests passed, the hooks now trust the output and override the exit code to zero. This worked because the marker is only written when the test command actually succeeds.

Then came a security issue. The hooks were checking for "RESULT: test PASSED" in the output, which could be spoofed by a malicious test command. The developer switched to using a special `VERIFY_EXIT` marker that only verify-phase.sh itself emits, making it impossible for test output to fake a passing result. They added a regression test to ensure this vulnerability wouldn't reappear. They also added diagnostic logging to capture debug info for post-mortem analysis if something goes wrong again.

Next, the developer discovered that when verification output exceeded 4000 characters, the hooks were keeping the first 4000 characters and discarding the rest. Since test failures and summaries appear at the end of output, this meant actual errors were getting hidden behind early passing tests. The fix flipped the truncation logic to keep the last 4000 characters instead of the first, ensuring failure messages stay visible.

In the afternoon, the developer consolidated configuration and documentation updates. They strengthened the adopting-new-technologies rule by making the `/research` skill invocation mandatory instead of optional, removing an escape hatch that was letting it be bypassed. They added a new rule requiring the `date` command to be run first whenever discussing dates, times, or scheduling topics to ensure deterministic information rather than relying on context or training data. They also documented presentation slide preferences and Quarto/Reveal.js capabilities in new rule files, and added a mandatory requirement to use the `/write-docs` command when writing documentation.

The day ended with refinement work on the global CLAUDE.md file. The developer removed a duplicate readability rule, narrowed the `/write-docs` requirement to apply only to user-facing documentation like READMEs (excluding CLAUDE.md and rule files themselves), and added a new "Conflict Resolution" section establishing that project-level rules take precedence over global ones.

## Key Decisions

- **Replace command output capture from `$()` pipes to temp file redirection** — Shell command substitution was producing incorrect exit codes with large output; temp files decouple output capture from exit code capture.

- **Remove `pipefail` from verify-phase.sh** — The flag was interfering with eval'd commands that contain implicit pipelines like npm process trees.

- **Implement output-based override for corrupted exit codes** — When process exit code is non-zero but output contains "RESULT: <phase> PASSED", trust the output and override exit code to 0. Belt-and-suspenders approach that provides immediate stability while the underlying exit code corruption issue remains unresolved.

- **Switch override check from RESULT marker to VERIFY_EXIT marker** — RESULT output can be spoofed by test commands; VERIFY_EXIT is only emitted by verify-phase.sh, preventing false overrides.

- **Reverse truncation direction to preserve test failure output** — Keep the last 4000 characters instead of the first, since test failures and summaries appear at the end of output.

- **Add diagnostic logging to hook scripts** — Persist debug info to `/tmp/verify-hook-debug.txt` after each hook run for post-mortem analysis without requiring reproduction.

- **Strengthen `/research` skill invocation requirement** — Made the requirement mandatory for adopting new technologies, removing the escape hatch that offered web search as an alternative.

- **Add deterministic date/time check before discussing scheduling topics** — Run `date` command first to get ground truth instead of relying on context or training data.

- **Establish mandatory `/write-docs` rule for documentation writing** — Enforce consistent documentation review process across user-facing docs (READMEs, guides), but exclude CLAUDE.md and rule files.
