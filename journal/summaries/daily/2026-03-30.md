# Daily Summary — 2026-03-30

## Narrative

The developer spent the morning working on two interconnected PRDs that emerged from exploring Claude Code's plugin architecture and security model.

It started with a question from a friend: can Claude Code jailbreak its own hooks? That sparked research into the security properties of different enforcement mechanisms. The developer discovered that while Claude Code hooks are more reliable than instructions in CLAUDE.md, they can still be behaviorally circumvented through creative command routing and regex gaps. Native git hooks, by contrast, run inside the git process itself with no alternative path—the agent would need to bypass git entirely to circumvent them. This became the foundation for PRD #47: migrating six git-related hooks from Claude Code's tool-call interception layer to native git hooks for deterministic, process-level enforcement.

The migration strategy was carefully designed around the developer's actual environment. They discovered their corporate laptop has a globally-set `core.hooksPath` pointing to Datadog's enterprise hook system, which initially looked like a blocker. Instead, the developer incorporated it into the architecture—Datadog's hooks dispatch to local `.git/hooks/` via a `run-local-hooks` mechanism, so the new native hooks will work within that existing system without conflicts. The developer phased the migration into two milestones: four "clean" hooks first (commit message checks, branch protection, progress tracking, test tiers), then two "complex" hooks (build/typecheck/lint verification and security checks). Two hooks that intercept GitHub CLI commands (`gh pr create` and `gh pr merge`) stay as Claude Code hooks since there's no git hook equivalent. A critical requirement emerged: every hook must print smart, actionable error messages that both humans and Claude can understand and act on. The developer made this a hard requirement in every milestone's success criteria, not just a nice-to-have.

The second major work stream came from the developer's desire to upgrade their frequently-used skills by studying how mature Claude Code plugins were constructed. They requested Anki cards about plugin architecture patterns with very specific guidance: each card must be completely self-contained—"EACH CARD IS AN ISLAND"—meaning no card should assume the reader had seen any other card or remembered the original conversation. They pushed back on vague references in drafts, insisting that "The Code Review plugin" needed to be "Anthropic's Code Review plugin for Claude Code" with full context every time. This principle revealed that the `/anki` skill itself needed updating. The developer formalized the "EACH CARD IS AN ISLAND" rule in the skill definition's "Future-Self Accessibility" section, adding explicit guidance about fully qualifying references, avoiding project-internal labels like PRD numbers, and naming the repository explicitly in every card.

The developer then created PRD #48 outlining six milestones to upgrade five existing skills with patterns discovered from studying three mature plugins: Anthropic's Skill Creator, Anthropic's Code Review plugin, and Jesse Vincent's Superpowers plugin. The research identified six cross-cutting patterns—confidence scoring over binary verdicts, multi-agent specialization, phased workflows with explicit gates, blind comparison for A/B decisions, evidence-based assertions, and progressive disclosure. Each milestone focused on one skill and a specific pattern: formalized decision gates for `/research`, progressive disclosure and severity scoring for `/write-prompt`, structured error capture for `/verify`, card-level confidence scoring for `/anki`, and a broken docs detection phase for `/write-docs`. The final milestone was a cross-skill consistency review.

At the very end of the day, the developer caught and fixed an inconsistency in the research documentation. The confidence scoring scale for the `/anki` skill had been documented as a single 0-10 scale, but the PRD specified a three-dimensional approach where each dimension is scored 1-5 (maximum 15 total), with cards below 9 flagged for review. The developer corrected the research doc to align with the PRD specification.

## Key Decisions

- **Migrate git-related hooks from Claude Code to native git hooks for deterministic process-level enforcement** — Claude Code hooks can be behaviorally circumvented through creative command routing and regex gaps; native git hooks run inside the git process itself with no alternative path. Defense-in-depth model: native git hooks as deterministic floor, CLAUDE.md as guidance layer. Tradeoff: removing Claude Code hooks eliminates `additionalContext` JSON guidance, but smart error messages printed by git hooks provide equivalent AI-readable output.

- **Phase migration into two milestones — clean four hooks first, complex two hooks second** — Clean hooks map directly to git events with minimal dependencies; complex hooks depend on shared infrastructure. Phasing reduces risk while proving the pattern before tackling high-value enforcement hooks.

- **Implement smart, actionable error messages as hard requirement in every hook** — Every migrated hook must print clear messages that both humans and Claude can understand and act on. Messages must explain what went wrong and how to fix it.

- **Use integration tests against real temporary git repositories** — Hook logic is simple enough that unit tests add maintenance cost without catching different bugs than integration tests. Integration tests directly validate the thing that matters: does git actually block the operation?

- **Install native git hooks into `.git/hooks/` per repository via bootstrap script** — Hooks live in `claude-config/hooks/git/` as source of truth; bootstrap script installs via symlinks. Datadog's `run-local-hooks` shim automatically dispatches local hooks before corporate policy enforcement.

- **Archive migrated Claude Code hooks rather than delete** — Migrated hooks archived to `hooks/archive/claude-code/` with documentation explaining migration; removed from `settings.json` to prevent dual enforcement.

- **Keep two hooks as Claude Code hooks** — `gh pr create` and `gh pr merge` intercept GitHub CLI commands with no native git hook equivalent.

- **Formalize "EACH CARD IS AN ISLAND" rule in the /anki skill** — Every card must be completely self-contained with fully qualified references; no card should assume the reader has seen any other card or remembered the original conversation. Avoid project-internal labels like PRD numbers; name the repository explicitly in every card.

- **Upgrade five skills with plugin architecture patterns** — Extract design patterns from mature plugins (Skill Creator, Code Review plugin, Superpowers) and apply them across `/research`, `/write-prompt`, `/verify`, `/anki`, and `/write-docs` skills.
