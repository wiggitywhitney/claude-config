# Daily Summary — 2026-04-11

## Narrative

The developer spent the day working on PRD verification and automation, starting with the core feature and then fixing related tooling issues that surfaced along the way.

They began by adding a pre-PR verification step to the prd-done skill. This verification agent reads the full PRD and maps each success criterion to observable evidence in the codebase using a three-level check: Exists (artifact is present), Substantive (real content, not a stub), and Wired (connected to the running system). The developer had initially considered running verification at the end of each milestone, but shifted to running it once at the /prd-done step instead—this keeps costs tractable while providing a full picture of completed work. In the careful-mode version, gaps block PR creation and ask the user how to proceed. In the YOLO version, the agent implements any missing work and re-runs verification automatically before moving forward.

While discussing verification, the developer also explored a broader idea about syncing decision logs across open PRDs. They settled on a two-layer approach: a global CLAUDE.md rule that reminds Claude to cascade-evaluate downstream milestones whenever a PRD's Decision Log is updated, plus a PostToolUse hook that fires when a decision log row is added to catch updates that might happen outside active conversation context. The hook fires on any write or edit to files in the prds/ directory, outputs an advisory message, and always exits successfully since it's meant to remind rather than block. The developer accepted that this cascade evaluation might be imperfect—70% accuracy is better than 0%, and recoverable drift beats silent drift.

They implemented this with a new cascade-decision-check.sh hook, registered it in settings.json, documented it in the hooks reference, added the CLAUDE.md rule, and wrote a bats test suite with ten tests covering both positive cases (the hook fires correctly) and negative cases (it stays silent for non-PRD files). They also committed rules/otel-semconv-gotchas.md, a reference document that had been referenced in CLAUDE.md but never actually added to the repo, and added a Testing section to the project CLAUDE.md requiring bats coverage for all bash hooks and scripts.

Next, they cleaned up the PR by reverting an unintended model configuration change from sonnet to opus, strengthening the test assertions from fragile substring checks to proper JSON schema validation, and removing a date stamp from the OpenTelemetry documentation so it wouldn't become stale.

The developer then updated the bats-bash-testing documentation with findings from running the tests against bats v1.12 and v1.13. They added paths frontmatter scoping the rule to .bats files and test shell scripts, clarified how the `run` command handles function visibility (functions defined in the test file are visible without export, but export -f is needed when using `run bash -c`), documented that mocked functions can bleed into bats internals and the recommended workaround, added information about the `bats_pipe` helper for pipelines, and documented two version-specific bugs that were fixed in v1.13.0: output crosstalk between run calls and BATS_TEST_TIMEOUT not killing processes started via run.

Finally, the developer fixed two bugs in the CodeRabbit verification hook. First, the hook was only checking one of three GitHub API channels where CodeRabbit posts reviews—when the review landed in issue comments instead of pull reviews, the merge got blocked with a false error. They expanded the check to query all three channels: pull reviews, inline comments, and issue comments. Second, they discovered a pagination bug: the GitHub API's default behavior without `--paginate` only fetches 30 items per page, so PRs with more than 30 reviews or comments would silently miss CodeRabbit's review and incorrectly block the merge. They refactored the three separate API calls into a single function that adds the `--paginate` flag and uses per-item line counting to work around how pagination interacts with jq aggregation. They added detailed test coverage for all three channels plus edge cases, including a specific test for the previously broken issue-comments scenario.

## Key Decisions

- **Add pre-PR verification agent step to validate milestone objectives** — Verification fires at /prd-done step rather than per-milestone to keep cost acceptable while providing a holistic view of completed work. Agent maps each criterion to observable evidence using a three-level check (Exists → Substantive → Wired). YOLO version auto-fixes gaps; careful-mode surfaces gaps to user.

- **Add PostToolUse hook to advise cascade-evaluation of downstream milestones when PRD Decision Log is updated** — Hook fires on Write/Edit to prds/ files (excluding prds/done/), outputs advisory message, always exits successfully. Complements global CLAUDE.md rule which fires during active decision-making. Two-layer approach catches updates both in-flow and outside active awareness. Accepts imperfect cascade evaluation as better than zero detection.

- **Revert unintended model configuration change** — Model was accidentally changed from sonnet to opus in the working tree and needed to be reverted to maintain original configuration.

- **Replace fragile substring assertions with parsed JSON schema validation** — Substring checks were brittle; switched to parsing JSON and verifying actual hookSpecificOutput schema structure for stronger guarantees.

- **Remove date from OpenTelemetry verification note** — Date stamp would become stale; removing it keeps the note evergreen without requiring future updates.

- **Add paths frontmatter to bats-bash-testing.md** — Scopes the rule to .bats files and test shell scripts in tests/ directories per project requirements.

- **Document run command behavior with .bats-defined functions** — Functions defined in .bats file are visible without export -f; export -f only needed when bash -c spawns a new process.

- **Document mock function bleed-through into bats internals** — Mocking system commands can break bats exit traps; fix is to define mocks inside setup() or @test body and unset -f in teardown().

- **Expand CodeRabbit verification to check all three API channels** — Hook was only checking pull reviews; expanded to also check inline comments and issue comments so reviews landing in any channel are detected.

- **Add --paginate flag to CodeRabbit channel checks** — GitHub API without --paginate only fetches 30 items; added pagination and switched to per-item line counting to handle PRs with many reviews or comments.
