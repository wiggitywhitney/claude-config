# Daily Summary — 2026-04-14

## Narrative

The developer spent the day completing and refining the code-review plugin integration. They started by wiring the plugin into the standard PR workflow, discovering along the way that user-scoped plugin installs aren't globally accessible across repos. Rather than trying to install the plugin everywhere, they followed the existing pattern used for other global skills: creating the skill file in the claude-config repo and symlinking it to `~/.claude/skills/code-review`. They added `/code-review` steps to both versions of the prd-done workflow, placed immediately after PR creation, and updated the git-workflow and hooks-reference documentation to clarify when the plugin runs and what output to expect.

A key clarification came early: the eligibility check needed to explicitly distinguish Claude Code reviews from CodeRabbit reviews, since the Haiku agent was misinterpreting "a code review from you" as any automated reviewer and skipping reviews when CodeRabbit had already commented. They also discovered that the confidence scoring rubric was ambiguous about whether scores were discrete or continuous, which unintentionally created a strict threshold. Both issues were corrected in the local skill file.

The developer marked PRD #65 complete and archived it, then spent the rest of the day iterating on the skill based on CodeRabbit feedback. They fixed several issues: added git history commands to the allowed-tools list so the agent could read blame and log; fixed the no-issues template to include a commit SHA link so the eligibility check would work correctly; and standardized placeholder notation to angle brackets for consistency. They also clarified the prd-done workflow to gate only the merge on `/code-review` findings, allowing CodeRabbit timer and Anki capture steps to proceed in parallel.

Later, they discovered a control flow bug where PRs with no issues found would silently stop before posting the result, preventing the eligibility marker from being posted. They fixed the routing so the no-issues path reaches the posting step.

The final major change came mid-afternoon: they replaced the hard confidence cutoff (80) with a two-tier triage system. Instead of silently filtering out findings below 80, findings scoring 50 or higher are now kept and separated into high-confidence (≥80) and medium-confidence (50–79) tiers. Each finding gets assigned a disposition—Fix (required before merge) or Skip (real but not worth addressing)—based on a clear rubric: would a senior engineer require this addressed before approving? The output format changed to structured tables grouped by confidence tier, making it easier to scan and prioritize. During this work, they decided against a Defer option because it lacked a tracking mechanism—deferring without infrastructure to follow up was just noise. They finished with a couple of small fixes: adding a Fix example to the medium-confidence template section to prevent bias toward Skip decisions, and removing redundant wording from the format instruction.

## Key Decisions

- **Wire `/code-review` plugin into standard PR workflow via skill symlink pattern** — User-scoped plugin installs aren't globally accessible; followed existing pattern used by anki and research by creating `.claude/skills/code-review/SKILL.md` in claude-config with plugin content and symlinking it. `/code-review` and CodeRabbit find different issue classes, so both should run on every PR for complementary coverage. Added `/code-review` step to prd-done skills immediately after PR creation with instruction to address findings at ≥80 confidence before continuing. Skills are where procedural instructions go; updating only rules files wouldn't reliably trigger the behavior.

- **Clarify eligibility check to distinguish Claude Code reviews from CodeRabbit reviews** — Haiku agent was misinterpreting "a code review from you" as any automated reviewer. Eligibility check now explicitly looks for Claude's "### Code review" comment signature.

- **Clarify confidence scoring is continuous (0–100), not discrete** — Rubric listed example scores but the text was ambiguous about whether only those values were acceptable. Clarified agents may use any integer within the range.

- **Expand allowed-tools to include git history commands** — Agent instructions referenced reading git blame and history, but git log/blame/show weren't in the allowlist. Added them to prevent requiring manual approval for each invocation.

- **Fix no-issues template to include HEAD SHA commit link for eligibility checks** — Clean PR reviews had no commit link, causing the eligibility check to always see "no SHA match" and re-run the full review. Added a commit link with the reviewed SHA.

- **Standardize placeholder notation to angle-bracket convention** — No-issues template used literal strings like `HEAD_SHA_PLACEHOLDER` instead of `<angle-bracket>` notation. Converted to `<full-sha>`, `<owner>`, and `<repo>` with explicit instruction for substitution.

- **Clarify `/code-review` gate timing in prd-done workflow** — "Before continuing" was ambiguous and could block CodeRabbit timer and Anki capture steps. Changed to "before merging" to clarify only the merge is gated on `/code-review` findings.

- **Add `/code-review` step to issue-juggling workflow** — Agents following issue-juggling.md would skip `/code-review` entirely without explicit instruction. Added step to run immediately after PR creation.

- **Replace hard confidence cutoff (80) with two-tier triage system** — Silent filtering prevented visibility into moderate-confidence findings. Two-tier approach surfaces findings ≥50 as either high confidence (≥80) or medium confidence (50–79), with automatic rubric application for Fix/Skip dispositions. Findings below 50 remain filtered out.

- **Establish Fix vs Skip distinction rubric** — Fix means a senior engineer would require it addressed before approving; Skip means the suggestion misunderstands code or complexity outweighs benefit. Removed Defer option due to lack of tracking mechanism in PR comments.

- **Restructure PR comment output as tiered table format** — Replace numbered list with separate tables for high-confidence and medium-confidence findings, each with score, disposition, and reasoning. Omit sections if no findings exist in that tier.
