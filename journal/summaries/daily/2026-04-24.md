## Narrative

The developer spent the day building and refining a progress tracking system for pull requests, tackling three separate issues in a single initial commit to minimize code review overhead.

The core work centered on a new pre-push hook called `progress-md-pr.sh` that enforces PROGRESS.md updates whenever code changes are pushed. The hook uses Claude to auto-draft entries from commit messages in interactive environments, then prompts the developer to accept, edit, or skip. In CI or non-interactive contexts, it just warns and lets the push through. The developer also rewrote Step 8.9 in the `/prd-update-progress` skill from a self-assessment checklist into an enforcement-based workflow with five concrete actions—scanning for undocumented decisions, verifying PROGRESS.md reflects today's changes, tracking open questions, ensuring the next task's entry point is clear, and documenting workarounds. This shift closed a loophole where the AI could self-assess without actually completing work.

A third issue in that initial commit fixed a portability problem in the `/continue` skill where macOS-only `date -v-1d` syntax wasn't working on Linux. The final commit caught another instance of the three-dot range bug in the draft commit message log.

## Key Decisions

- **Replace macOS-only date command with cross-platform Python fallback** — The `/continue` skill was using `date -v-1d`, which doesn't work on Linux. Python works universally; platform-specific alternatives are documented for environments without Python 3.

- **Rewrite Step 8.9 in prd-update-progress from self-assessment to enforcement-based actions** — The original "answer honestly" framing created a loophole where the AI could self-assess without completing work. Each of the five items (decisions, PROGRESS.md, open questions, next task entry point, workarounds) is now an action to complete with a testable success criterion.

- **Add pre-push hook that enforces PROGRESS.md entries** — Blocks pushes on branches with commits but no PROGRESS.md changes. Uses Claude to auto-draft from commit messages in interactive mode; non-interactive environments (CI) get a warning and allow the push through.

- **Use two-dot git range instead of three-dot for commit counting** — Three-dot ranges count commits reachable from either side, causing false positives. Two-dot ranges correctly count only commits unique to the branch being pushed.

- **Scope PROGRESS.md entry insertion to the [Unreleased] section** — Prevents accidental insertion into historical release sections. Creates the `### Added` subsection if needed, falls back to appending at the end if no [Unreleased] section exists.

- **Add timeout to Claude draft call** — Prevents hanging invocations from blocking pushes indefinitely.

- **Use regex for PROGRESS.md section matching** — Handles variable whitespace after section headings instead of relying on exact string matches.

- **Add fallback for timeout command availability** — Homebrew's coreutils on macOS provides `gtimeout` instead of `timeout`. Try `timeout` first, then `gtimeout`, then run without a timeout wrapper.

- **Validate auto-draft content before accepting** — Block the accept path if the draft is empty or whitespace-only to prevent empty entries from being committed.

- **Check git operations succeed before reporting success** — Only print the success message if both `git add` and `git commit` actually succeed; otherwise, tell the user to commit manually.
