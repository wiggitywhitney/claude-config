# Daily Summary — 2026-05-13

## Narrative

The developer spent the day building and refining a pair of related hooks for the AI agent system, addressing two distinct problems with how context flows during planning and handoff moments.

They started by extending the existing `suggest-write-prompt` hook, which reminds the AI to use the `/write-prompt` command when working on prompt-related files. The hook was originally firing only for `SKILL.md`, `CLAUDE.md`, and GitHub issue creation, but the developer realized it was missing a lot of files that agents actually work on—PRDs, rules files, and system prompts saved under non-standard names like `system-prompt.md` or `agent-spec.md`. They expanded the hook's pattern matching to catch all three categories and wrote a 22-test suite to verify the hook fired correctly for the new file types while staying silent on everything else.

While working on that, a separate issue came up in conversation: context was getting lost during planning handoffs. The developer would plan work with an AI instance, that work would end up in a PRD or GitHub issue, but not all the important conversation context would make it into the document. Then they'd have to repeat the planning conversation with the next AI instance. This felt like a different problem than the implementation handoff that the existing hooks address—it's about completeness of planning context, not quality of prompts. So instead of mixing it into the expanded `suggest-write-prompt` hook, they created a new hook called `suggest-planning-handoff` that fires only at actual handoff moments: when `gh issue create` succeeds or when a new PRD file gets written. When it fires, it asks three concrete questions: were decisions from the conversation captured, were open questions captured, and could a future AI act on the document alone? They built an 18-test suite for this one too, covering the trigger cases and all the scenarios where it should stay quiet (like editing an existing PRD mid-session, or other gh subcommands).

Later in the day, the developer discovered a bug in both hooks: they were using glob patterns like `*/prds/*` and `*/rules/*` that required a parent directory, so relative paths like `prds/93-feature.md` weren't matching. They fixed both scripts by adding simpler patterns (`prds/*` and `rules/*`) to catch relative paths alongside the absolute ones, and added test cases to verify the fix worked.

Finally, they merged the feature branch back into main, bringing both hooks together with all their tests and documentation updates.

## Key Decisions

- **Extend suggest-write-prompt hook to fire for PRD files, rules files, and non-standard prompt filenames.** The hook was missing file types that agents interact with regularly. Expansion is straightforward—just additional pattern matching on file paths and names.

- **Create a separate planning-handoff hook instead of mixing into suggest-write-prompt.** Planning handoff (completeness of context) and implementation handoff (quality of prompts) are different concerns. A separate hook avoids noise and keeps each hook focused.

- **Fire planning-handoff hook only on new file creation, not edits.** Using the Write tool (creates) versus Edit tool (modifies) distinction prevents the hook from firing repeatedly during mid-session PRD refinement, avoiding reminder fatigue and duplication with existing hooks like cascade-decision-check.

- **Register planning-handoff hook in both Bash and Write PostToolUse matchers.** The hook needs to catch both `gh issue create` (via Bash) and new PRD file creation (via Write), so it's registered in both matcher sections.

- **Fix path matching in both hooks to handle relative paths.** Glob patterns like `*/prds/*` missed relative paths like `prds/93-feature.md`. Added simpler patterns (`prds/*`, `rules/*`) to catch both cases.
