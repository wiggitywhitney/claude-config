## Narrative

The developer spent the day building out the autonomous PRD execution system, starting with a critical research phase and then implementing the detection and coordination mechanisms.

The work began with M4 — a SessionStart hook that detects when an autonomous Ralph loop is running in the current repo and warns interactive sessions before they collide with it. Before diving into implementation, the developer paused to validate the design against Claude's actual behavior. They had concerns about whether `claude -p` headless sessions would fire SessionStart hooks the same way interactive sessions do, and whether file markers were really the best coordination mechanism. The research confirmed that `claude -p` children do fire all SessionStart hooks identically to interactive sessions (only `--bare` mode suppresses them, but that also strips the full toolchain that autonomous children need). This finding was critical — it meant the hook would fire in every child the orchestrator spawned unless explicitly suppressed. So the developer implemented an environment variable suppression pattern: the orchestrator sets `AUTONOMOUS_PRD_CHILD=1` before each `claude -p` spawn, and the hook checks for this env var at the top and exits silently if set. This prevents children from seeing the human-facing "don't make edits" warning in their own context, which could confuse them into not doing their assigned work. The hook itself reads `.claude/autonomous-run.local` (written by the orchestrator on startup), checks whether the PID it contains is still alive using `kill -0`, and surfaces either a "running" warning or a "stale marker" warning to stderr. The developer wrote a 7-test bats suite covering all four states: child suppression, absent marker, live PID, and dead PID. They also updated the M4 section of the PRD to reflect this new scope and cascaded the decision to M6 and M7 so documentation and validation account for the suppression mechanism.

Next, the developer updated the autonomous PRD execution system to handle headless mode correctly. The two YOLO skills that drive the PRD loop — `prd-next` and `prd-update-progress` — were updated to detect whether they're running in autonomous mode (by checking for the `.claude/autonomous-run.local` marker file) and branch accordingly. In autonomous mode, they stop invoking tools and let the orchestrator handle the next iteration. In interactive mode, they preserve the existing behavior of asking the user to run `/clear` manually. The developer also created a new rule file documenting the full lifecycle of pause markers: the two types (child-written for decisions that need human input, orchestrator-written for iteration caps), how the orchestrator detects them before spawning new children, the resume protocol, and the failure mode if you remove a marker without fixing what caused it. They cleaned up dead code from `auto-reanchor.sh` that was checking for a file that never existed under this PRD's design, along with the corresponding tests.

Finally, the developer removed an unused array from a test file that was flagged by a code review tool — it had been defined to document permission entries but was never actually referenced in any of the tests.

## Key Decisions

- **SessionStart hook suppression in `claude -p` child sessions via environment variable** — `claude -p` headless sessions fire all SessionStart hooks identically to interactive sessions. Without suppression, each child's `additionalContext` would contain the human-facing warning, which could confuse children into not doing their assigned work. Environment variable is lightweight, forwarded automatically by the shell, and requires no file or socket coordination. Distinct from the file-marker approach used for YOLO skill detection — env var is hook suppression only, not mode detection.

- **Four-state detection logic for autonomous run marker** — Child session suppression (if `AUTONOMOUS_PRD_CHILD` is set, exit silently), marker absent (exit silently), marker present with live PID (surface "running" warning to stderr), marker present with dead PID (surface "stale marker" warning). PID liveness checked via `kill -0`. Stale markers are not auto-removed — surface and let user decide.

- **Orchestrator sets `AUTONOMOUS_PRD_CHILD=1` before each `claude -p` spawn** — One-line change at each spawn site. Applied to both milestone spawn and final `/prd-done` spawn. Prevents children from triggering the collision warning in their own `additionalContext`.

- **SessionStart hook outputs warnings to stderr for `additionalContext` integration** — Warnings written to stderr so Claude Code picks them up as `additionalContext`. Follows the pattern established by `scripts/auto-reanchor.sh`. Always exits 0 (advisory, not a gate).

- **Marker file format and parsing approach** — Marker file contains `PID=`, `STARTED=`, and `PRD=` fields on separate lines. Parsed using `grep` and `cut` rather than sourcing to avoid executing untrusted file content. Marker path derived from `git rev-parse --show-toplevel` to work across all repos.

- **Test coverage for all four detection states** — Tests cover child suppression, absent marker, live PID, and dead PID. Tests use real temporary git repositories, same pattern as existing hook tests in the project.

- **YOLO skills detect autonomous mode and branch on behavior** — `prd-next` and `prd-update-progress` check for `.claude/autonomous-run.local` marker file. In autonomous mode, stop invoking tools and let orchestrator handle next iteration. In interactive mode, preserve existing behavior of asking user to run `/clear` manually.
