# Weekly Summary — 2026-W12

## Week in Review

The developer tackled two distinct problem areas this week: decision propagation and configuration management on Monday, then moved to git hook reliability on Friday.

On Monday, they identified that decisions were getting logged but weren't reaching the agents actually implementing work. The fix was to add a propagation step to the prd-update-decisions skill that scans incomplete milestones and updates any affected by recent decisions, with each update referencing the decision number to keep the log as the single source of truth. They also added decision awareness nudges to the prd-update-progress and research skills so decisions get recorded while context is fresh.

The same day, they tackled a growing problem: the global CLAUDE.md file had ballooned to 172 lines, and every line gets loaded into every conversation across all projects. They created a refactoring plan with detailed analysis of what could be extracted versus what needed to stay, then executed it by moving verbose procedural sections into six new rule files while keeping short, domain-specific sections inline. The result shrunk CLAUDE.md from 172 lines to 99 lines. They also standardized reference paths to use symlink-based patterns and added guidance for running CodeRabbit reviews manually.

Friday's work focused on a critical bug in the git hook system. Large test suites (1700+ tests producing 20KB+ output) were falsely blocking commits even when all tests passed. The root cause was that shell command substitution using `$()` was producing incorrect exit codes when capturing large amounts of output. They fixed this by replacing pipe capture with temporary file redirection across three hook scripts and removing the `pipefail` flag that was interfering with eval'd commands.

But fixing that wasn't enough—hooks were still blocking valid commits due to external processes like GitLab credential helpers corrupting exit codes during execution. Rather than trying to prevent the corruption, they added a defensive override: when exit codes are non-zero but the output contains a special `VERIFY_EXIT` marker (which only verify-phase.sh emits), the hooks trust the output and override the exit code to zero. They also fixed a truncation bug that was keeping the first 4000 characters of output instead of the last, which meant test failures appearing at the end were getting hidden.

The afternoon wrapped up with configuration and documentation updates. They made the `/research` skill mandatory for adopting new technologies, added a rule requiring the `date` command before discussing scheduling topics, documented presentation preferences, and made `/write-docs` mandatory for user-facing documentation (while excluding CLAUDE.md and rule files themselves).

## Highlights

- Fixed critical git hook bug blocking valid commits by replacing command substitution pipes with temp file redirection and implementing a defensive exit code override using a special marker.
- Refactored global CLAUDE.md from 172 lines to 99 lines by extracting verbose sections into six new rule files while preserving all original wording, reducing token cost across all conversations.
- Added decision propagation to prd-update-decisions skill so decisions automatically flow to affected milestones, and added decision awareness nudges to prd-update-progress and research skills.

## Patterns

- **Git hook reliability was the focus of Friday's work.** The developer discovered and fixed multiple interconnected issues: exit code corruption from command substitution, external process interference, and output truncation logic that was hiding errors. Each fix built on the previous one, suggesting the hooks had accumulated fragility over time.
- **Configuration and documentation consolidation appeared on both days.** Monday involved extracting CLAUDE.md sections into rule files; Friday involved strengthening rules and adding new ones. This reflects ongoing work to keep global configuration lean and explicit.
- **The developer used defensive programming patterns to work around unresolved root causes.** The exit code override masks the underlying corruption issue rather than fixing it, and diagnostic logging was added for post-mortem analysis—suggesting they're stabilizing the system while acknowledging deeper issues remain.
