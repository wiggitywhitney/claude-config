# Weekly Summary — 2026-W14

## Week in Review

The developer spent the week building out a defense-in-depth security model for git hooks and carefully upgrading five frequently-used skills with patterns extracted from mature Claude Code plugins.

The week started with research into Claude Code's security properties. The developer discovered that while Claude Code hooks can be behaviorally circumvented through creative routing and regex gaps, native git hooks run inside the git process itself with no alternative path. This insight drove PRD #47: migrating six git-related hooks from Claude Code's tool-call interception to native git hooks for deterministic, process-level enforcement. Rather than treating the corporate Datadog hook system as a blocker, the developer incorporated it into the architecture—Datadog's hooks dispatch to local `.git/hooks/` automatically, so the new native hooks will integrate seamlessly. The migration got phased into two milestones (four "clean" hooks first, then two "complex" ones), with a hard requirement that every hook print smart, actionable error messages both humans and Claude could understand and act on. Two GitHub CLI hooks stayed as Claude Code hooks since git has no equivalent.

In parallel, the developer formalized a principle for skill upgrades: "EACH CARD IS AN ISLAND." Every flashcard must be completely self-contained with fully qualified references—no card should assume the reader had seen any other card or remembered the original conversation. This revealed that the `/anki` skill definition itself needed updating. The developer pushed back on vague references in drafts and added explicit guidance to the skill's "Future-Self Accessibility" section about naming the repository explicitly in every card and avoiding project-internal labels like PRD numbers.

This led to PRD #48, which outlined upgrading five skills (`/research`, `/write-prompt`, `/verify`, `/anki`, and `/write-docs`) with six cross-cutting patterns discovered from studying three mature plugins: confidence scoring over binary verdicts, multi-agent specialization, phased workflows with explicit gates, blind comparison for A/B decisions, evidence-based assertions, and progressive disclosure. Each skill got a targeted upgrade.

The `/research` skill got explicit decision gates—three validation checkpoints ensuring question specificity, multi-source coverage, and explicit handling of contradictions rather than silent resolution. A "Conflicting Findings" section now surfaces disagreements in the output template.

The `/write-prompt` skill got severity scoring for its anti-pattern review system, separating high-severity issues that break things from medium ones affecting efficiency and low ones that are style preferences. The review output reorganized into three tiered sections, and a new gate skips certain validation checks when reviewing Claude Code skills instead of raw API prompts.

The `/verify` skill was upgraded to emit structured error transcripts when verification phases fail. Instead of dumping raw output, the script now captures command output and generates a JSON object containing the phase name, command, exit code, timestamp, and the last 20 lines of output. The JSON gets printed with a `VERIFY_ERROR_CONTEXT:` marker and saved to a temp file so the system can detect if the same phase keeps failing across restart cycles. The SKILL.md documentation was updated to explain how to parse this JSON and produce targeted error summaries with concrete suggested fixes rather than generic troubleshooting advice.

The `/anki` and `/anki-yolo` skills got a three-dimension quality scoring system rating each card on memory anchor clarity, future-self accessibility, and concept-versus-detail balance. Cards scoring below 9 out of 15 are automatically rewritten once, targeting their weakest dimensions, then re-scored. If they still fall short, they're accepted with a note explaining why. Story-First Framing was wired in as an explicit step before card structuring. Every card now requires at least one hierarchical tag using Anki's `::` convention to enable filtered study sessions. The developer implemented a glossary index system that tracks which technologies already have foundational "What is X?" cards, and the skills now automatically queue missing terms as cards without requiring user deferral. The developer also researched how images work in their Anki workflow and pinned down technical decisions for an image bank milestone: `![[filename.png]]` syntax works correctly through Obsidian's vault index, images should be 800px wide PNG format, they must live inside the vault at `~/Documents/Journal/anki/images/bank/`, and filenames should follow a `concept-name-bank.png` convention to prevent collisions.

The `/write-docs` skill got a Broken Docs Detection phase that tests existing documentation before writing anything new by scanning for existing docs, executing all code examples, and producing a findings table. The skill adopted a "write-directly" pattern that removes approval gates between sections—the skill writes immediately and the user interrupts if changes are needed. Enforcement rules prevent command-skipping and environment give-up: commands are never skipped just because they seem simple, missing tools get installed rather than handed back to the user, and error cases must be documented alongside success paths. Commands now use full-form flags like `--filename` instead of `-f` to be self-documenting. The `/research` skill was integrated at two points: during Phase 1 when the documentation target involves new technology, and during Phase 2 when version drift looks like the culprit. A bug in the PRD loop continuation script got fixed—it was routing PRDs to completion prematurely because it only checked checkbox-style task lists, not heading-based milestones.

By mid-week, the developer had completed PRD #48's skill upgrades and turned attention to persistence. PRD #51 added persistence to the `/research` skill so research output doesn't disappear when the conversation ends. Research now writes to markdown files organized by topic under `docs/research/`, maintains a lightweight index to track what's been researched, and checks that index before starting new searches to avoid duplicate work. The developer worked through design decisions around preventing lazy updates, settling on requiring an explicit changelog entry for every modification that names what's being removed and added. They also decided against date prefixes in filenames for better discoverability and chose a simple one-line-per-file index format.

PRD #52 added prompt quality review to the prd-create skills. The insight was that PRD milestones function as de-facto instructions for future AI implementors, so they need prompt-quality review before being committed. Both the main and YOLO variants of the prd-create skill now include a `/write-prompt` review step after milestone definition. A PostToolUse hook was implemented to suggest running `/write-prompt` when skill documentation files are edited, using a test-driven approach.

The developer then implemented the research persistence feature, using `git rev-parse` to detect the repository root and converting research topics into filename slugs. The implementation writes structured markdown files with date headers, update logs, findings, and sources. An update protocol requires reading existing files in full before modification to prevent silent overwrites, with explicit changelog entries documenting what changed. The developer also added Phase 1 Step 5 to check the prior research index before conducting web searches, allowing the skill to surface existing findings and ask whether to extend, update, or accept them rather than duplicating effort.

Throughout the week, the developer engaged in thoughtful design discussions about avoiding AI laziness (particularly around research updates), preventing noise in advisory systems, and ensuring transparency in how research evolves across sessions. Code review feedback led to clarifications around skipping destructive commands during testing, refining what "materially different" output means, and distinguishing between project-level dependencies (which install automatically) and system-level tools (which require user confirmation).

## Patterns

- **Skill upgrades clustered around a single insight**: All five skills upgraded this week (PRD #48) were driven by extracting six cross-cutting patterns from three mature plugins—confidence scoring, multi-agent specialization, phased workflows, blind comparison, evidence-based assertions, and progressive disclosure. Rather than upgrading skills in isolation, the developer identified a unifying architecture and applied it carefully.

- **Persistence as a recurring theme**: Three separate PRDs this week (the git hook migration, research persistence, and glossary indexing) all centered on making ephemeral work durable—whether through native git hooks, markdown files, or index files. The developer recognized that work disappearing when conversations end was a pattern worth fixing across multiple systems.

- **Design-before-implementation discipline**: The developer completed two full PRDs (47 and 48) with detailed milestones and success criteria before starting implementation. This upfront design work prevented false starts and clarified dependencies—for example, discovering that the corporate Datadog hook system could be incorporated rather than worked around.

- **Code review feedback driving clarification**: Throughout the week, code review comments led to concrete improvements: fixing image selection logic, clarifying answer-reveal rules, distinguishing project-level from system-level dependencies, and refining what "materially different" output means. Rather than treating feedback as obstacles, the developer used it to tighten specifications.

- **Recursive skill improvement**: The developer upgraded `/write-prompt` to review anti-patterns in other skills, then used that same `/write-prompt` skill to review PRD milestones, creating a feedback loop where improvements to one skill immediately benefit others.
